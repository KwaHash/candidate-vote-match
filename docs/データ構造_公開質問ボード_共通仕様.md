# 公開質問ボード データ構造 共通仕様（3サイト一致版）

国民 `next-vote-match` / 運営 `admin-vote-match` / 政治家 `candidate-vote-match` の3アプリが、
**同一Supabaseの同一テーブル**を共有して「公開質問ボード」を動かすための**正準（canonical）データ構造**を定義する。

各アプリのプロトは現在それぞれの localStorage 形を持つ（型がズレている）。本書の正準スキーマに揃えることで、
`_store.ts` を差し替えるだけで3アプリが同じデータを読み書きできるようにする。

---

## 0. 現状の不一致（要統一ポイント）

| 項目 | 国民(citizen) | 運営(admin) | 政治家(candidate) | → 正準 |
|---|---|---|---|---|
| 質問ID | `id: string` `'q-seed-1'` | `id: number` | `id: string`（citizenと一致済） | **uuid（text id可）。3アプリ同一値** |
| 選挙 | `electionId` + `electionName` | `election`（名称のみ） | 定数 `ELECTION`（名称） | **`election_id: text`（名称はjoin）** |
| テーマ | `category`（表示名） | `category`（表示名） | `category`（表示名） | **`theme_key: text`（表示名はTHEMESで解決）** |
| 投票数 | `voteCount` | `votes` | `votes`（参照） | **`vote_count: int`（実体はvotes表）** |
| 上位質問 | status `'上位質問入り'` | `top: boolean` | — | **statusから導出（§4）** |
| 候補者 | `candidate`（氏名） | `answers` の key=氏名 | `ME`（氏名） | **`candidate_id: uuid`（表示名はjoin）** |
| 回答状態 | 回答済み/未回答/辞退/対象外 | 同左 | 未回答/下書き/提出済み | **6値に統合（§4）** |
| 回答の持ち方 | 質問に配列で内包 | `Record<氏名,状態>` | 質問ごとに1件 | **`candidate_question_answers` 表に正規化** |

## 1. ID・命名規約
- **`election_id: text`**：選挙の識別子。`ONGOING_ELECTIONS` / admin ロスター / 候補者側で同一値（例 `'suginami-chiji'`, `'osaka-chiji'`）。
- **`public_questions.id: uuid`**：質問の一意ID。**3アプリで同一値**を参照（プロトの `'q-seed-1'` 等は本番でuuidに置換、移行時は対応表を保持）。
- **`candidate_id: uuid`**：`candidates.id`（= `auth.users.id`）。表示名は `candidates.name` をjoin。
- DBは **snake_case**、プロトTSは **camelCase**。対応は §5。

## 2. 正準テーブル（Supabase / DDL）

```sql
-- 参照: 選挙・候補者（既存）
-- elections(id text pk, name text, region text, election_date date, ...)
-- candidates(id uuid pk, name text, ...)            -- 政治家側
-- election_candidates(election_id text, candidate_id uuid, ...)  -- 選挙×候補者

-- 公開質問
create table public_questions (
  id uuid primary key default gen_random_uuid(),
  election_id text not null references elections(id),
  theme_key text not null,                 -- 'kosodate' 等（表示名はTHEMESで解決）
  title text not null,
  body text,
  nickname text,                           -- 投稿者ニックネーム（任意）
  anonymous_key text,                      -- 匿名投稿者の識別（任意）
  vote_count int not null default 0,       -- 集計キャッシュ（実体は public_question_votes）
  status text not null default '運営確認中',-- question_status（§4）
  merged_into_question_id uuid references public_questions(id), -- 統合先（統合元に設定）
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

-- 投票（1人1票）
create table public_question_votes (
  id uuid primary key default gen_random_uuid(),
  question_id uuid not null references public_questions(id) on delete cascade,
  user_id uuid references auth.users,      -- ログイン投票
  anonymous_key text,                      -- 匿名投票（Cookie等）
  ip_hash text,                            -- 多重投票検知
  created_at timestamptz not null default now()
);
-- 1人1票（ログイン or 匿名キーのいずれか）
create unique index uq_vote_user on public_question_votes(question_id, user_id) where user_id is not null;
create unique index uq_vote_anon on public_question_votes(question_id, anonymous_key) where anonymous_key is not null;

-- 候補者の回答（全候補者×上位質問）
create table candidate_question_answers (
  id uuid primary key default gen_random_uuid(),
  election_id text not null references elections(id),
  question_id uuid not null references public_questions(id) on delete cascade,
  candidate_id uuid not null references candidates(id),
  answer_text text,
  source_url text,                         -- 公式リンク
  attachment_url text,                     -- 添付資料
  status text not null default '未回答',    -- answer_status（§4）
  answered_at date,
  reviewed_by_admin boolean not null default false,
  updated_at timestamptz not null default now(),
  unique(question_id, candidate_id)        -- 質問×候補者で1件
);
```

## 3. 表示名の解決（theme_key / election / candidate）
- **theme_key → 表示名**：3アプリ共通の `THEMES`（`kosodate`→「子育て・教育」…）。プロトは現状「表示名」を保存しているが、本番は `theme_key` を保存し表示時に解決する。
- **election_id → 選挙名/地域/投票日**：`elections` をjoin。
- **candidate_id → 候補者名/所属**：`candidates` / `election_candidates` をjoin。
- 回答比較表の「列（候補者）」は **`election_candidates` の候補者集合**を正とする（回答が無い候補者も列に出し「未回答」表示）。

## 4. 正準ステータス（enum）と状態機械

### 4-1. question_status（公開質問のライフサイクル・順序あり）
`運営確認中 → 公開中 → 上位質問入り → 候補者へ送付済み → 回答受付中 → 回答公開中`（＋ 終端 `非公開`）

| status | 意味 | 国民側の見え方 |
|---|---|---|
| 運営確認中 | 投稿直後・未公開 | 投稿者本人以外に出さない（投票不可） |
| 公開中 | 運営承認・一覧表示 | 表示・投票可 |
| 上位質問入り | 投票上位として運営選定 | 表示・投票可・「上位」表示 |
| 候補者へ送付済み | 送付文を全候補者へ送付 | 表示・投票可 |
| 回答受付中 | 回答募集中 | 表示・回答状況表示 |
| 回答公開中 | 回答を公開 | 比較表に回答を表示 |
| 非公開 | 却下／統合元 | 非表示 |

- **上位質問（is_top）は status から導出**：`status の順位 ≥「上位質問入り」 かつ ≠ 非公開`。
  → admin の `top: boolean` は本カラムを廃し、status で表現（管理UIは「上位に追加」=status を `上位質問入り` に上げる操作）。

```
[citizen投稿] → 運営確認中
運営確認中 ──admin公開──▶ 公開中 ──admin却下──▶ 非公開
公開中 ──citizen投票──▶ 公開中(vote_count++)
公開中 ──admin上位選定──▶ 上位質問入り ──admin送付──▶ 候補者へ送付済み
候補者へ送付済み ──回答募集──▶ 回答受付中 ──回答公開──▶ 回答公開中
（任意）──admin統合──▶ 非公開（merged_into_question_id を統合先に設定）
```

### 4-2. answer_status（候補者回答のライフサイクル）
`未回答 → 下書き → 提出済み（運営確認中） → 回答済み（公開）`（＋ `回答辞退` / `対象外`）

| status | 書く主体 | 国民側 比較表の表示 |
|---|---|---|
| 未回答 | 既定 | **未回答** |
| 下書き | 候補者 | 未回答（公開前） |
| 提出済み（運営確認中） | 候補者 | 未回答（公開前） |
| 回答済み（公開） | 運営admin（承認） | **回答済み** |
| 回答辞退 | 候補者/運営 | **回答辞退** |
| 対象外 | 運営 | **対象外** |

```
未回答 ──candidate下書き保存──▶ 下書き
下書き/未回答 ──candidate提出──▶ 提出済み（運営確認中）
提出済み（運営確認中） ──admin承認──▶ 回答済み（公開）
任意 ──candidate辞退──▶ 回答辞退 ／ 運営判断 ──▶ 対象外
```
- 国民側は **回答済み（公開）/回答辞退/対象外** をそのまま表示、それ以外は「未回答」。
- 既存プロトの `回答済み` は本番値 `回答済み（公開）` に対応（UIラベルは「回答済み」で可）。

## 6. 書き込み権限（RLS 方針）
- **public_questions**
  - insert：誰でも可（未ログイン含む）。ただし **status は必ず `運営確認中` 固定**（クライアントから他値を入れさせない）。
  - select：`status ∈ {公開中, 上位質問入り, 候補者へ送付済み, 回答受付中, 回答公開中}` は公開。`運営確認中/非公開` は admin のみ。
  - update/delete：**admin（運営）ロールのみ**（status遷移・theme/タイトル整え・統合）。
- **public_question_votes**
  - insert：誰でも可。対象質問が公開status のときのみ。unique制約で1人1票。update/delete 不可。
  - `vote_count` は votes の件数で更新（トリガ or アプリ）。
- **candidate_question_answers**
  - insert/update：候補者本人（`candidate_id = auth.uid()`）。設定可能 status は `{未回答, 下書き, 提出済み（運営確認中）, 回答辞退}`。
  - admin：`回答済み（公開）` / `対象外` への変更と `reviewed_by_admin=true`。
  - select：`status ∈ {回答済み（公開）, 回答辞退, 対象外}` は公開（比較表）。下書き/提出済みは本人＋adminのみ。

---

## 7. 集計と統合
- **vote_count**：`public_question_votes` の insert トリガで `+1`（または定期再集計）。
- **類似質問の統合**：admin が A,B,… を選び A に統合する場合
  1. B 等の `merged_into_question_id = A.id`、`status = 非公開`。
  2. B 等の votes を `update question_id = A`（または A.vote_count に件数を加算）。重複投票は unique 制約で吸収。
  3. B 等の回答（`candidate_question_answers`）は A に付け替えず破棄（または運営判断で移行）。UIには出さない。

## 8. プロト整合のための小修正（「完全一致」を実体化）
本番DBに乗せる前に、プロトの型を正準へ寄せる軽微修正（任意・推奨）。

1. **admin `QBoardItem`**：`id` を string 化し citizen の質問IDと一致／`election`→`election_id` 追加／`top` 廃止し status で表現／`body` 追加／`answers` の key を candidate_id（プロトは氏名でも可）。
2. **candidate `public-answers`**：各質問に `election_id` と `question_id`（citizenと同値）を保持／`ME`→ `candidate_id`／`AnsState` に `回答済み（公開）/回答辞退/対象外` を追加（提出後の表示用）／`attachment_url` を `source_url` と分離。
3. **citizen `_questions.ts`**：`category`→`theme_key`（表示はTHEMESで解決）／回答に `source_url`・`answered_at` を追加／`candidate`→`candidate_id`（表示名はjoin）。
- 共有定数（THEMES / question_status / answer_status / election_id 一覧）は **1ファイルに集約**し、3アプリで同一値を参照（モノレポ共有 or コピー同期）。

## 9. 受入条件（データ一致）
- [ ] 3アプリが同一の `public_questions.id`（質問ID）と `election_id` を参照する
- [ ] テーマは `theme_key`、選挙は `election_id`、候補者は `candidate_id` で保持（表示名はjoinで解決）
- [ ] question_status / answer_status が本書の値・順序・遷移に一致
- [ ] 上位質問は status から導出（`top` boolean を二重に持たない）
- [ ] 投票は 1人1票（unique制約）、`vote_count` は votes 由来
- [ ] 候補者回答は `candidate_question_answers` に正規化（質問×候補者で1件）
- [ ] RLS：質問の公開は admin、回答は本人＋adminの承認、未公開は非表示
- [ ] 統合で投票合算・統合元は非公開・`merged_into_question_id` 設定

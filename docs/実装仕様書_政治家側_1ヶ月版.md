# 実装仕様書（政治家向け candidate.seijiselect.jp / 1ヶ月版・MVP再設計）

完成済みプロトタイプ（`prototype` ブランチ）を流用して本実装するための仕様書です。
**初期リリースは機能を絞り込んだ MVP**。重い機能（実AI・寄付決済・法令対応・スクレイピング）はスコープ外。

- 対象リポジトリ: 本リポジトリ `candidate-vote-match`
- ベース: ブランチ `prototype` / タグ `proto-v1`
- 関連: 画面ごとの仕様=`PROTOTYPE_LOG.md`、全体構想=`BACKLOG.md`

---

## 1. コンセプト・MVP方針
- **「政治活動を、ひとつの画面で回す。」** ログイン後トップは「今日やること」。
- 機能を並べすぎない／初心者でも迷わない／入力は最小限から。
- **AIは「下書き→確認→編集→公開・送信」**。AIが勝手に公開・送信する機能は作らない。
- 政治・選挙・献金に関わるため、各所に**コンプライアンス注意文**を表示。

## 2. 進め方（重要：作り直さない）
1. 画面は完成済みプロトを流用（作り直さない）。
2. バックエンドは作らず **Supabase**（DB＋認証＋ファイル保存）。
3. デプロイは **Vercel**。
4. 中心作業=「**`app/prototype/_store.ts` の保存処理を Supabase に差し替え**」＋「**ログイン接続**」。

## 3. 画面構成（MVP・5メニュー）
ログイン後のメニューは5つに整理（プロト一覧 `/prototype` が再編済み）。

| # | メニュー | 画面 / ルート | 統合・備考 |
|---|---|---|---|
| 1 | **今日やること** | ダッシュボード `/prototype/dashboard` | 選挙活動タスクを統合。AIが「今日やること」を提案 |
| 2 | **政策をつくる** | 政策スタンス入力 `/prototype/policy-stance`、AI政策秘書 `/prototype/ai-secretary` | **AI政策室をAI政策秘書に統合**。Phase2で**1枚政策カード**・政策12項目を追加 |
| 3 | **発信をつくる** | AI政策秘書（発信）`/prototype/ai-secretary`、LINE・SNS発信 `/prototype/community` | 複雑なセグメント配信は後回し |
| 4 | **支援者を管理する** | 支援者CRM `/prototype/supporter-crm` | **支援募集を統合**。Phase3で「今日連絡すべき/お礼/フォロー漏れ」抽出・連絡文案 |
| 5 | **資金・成果を報告する** | 政策応援ページ `/prototype/crowdfunding`、定期献金管理 `/prototype/donation-manage`、支出・証憑（簡易）`/prototype/expenses`、寄付者への成果報告 `/prototype/donor-report` | **クラファン→「政策応援ページ」に改名**。献金は法務注意文必須 |

サブメニュー: プロフィール設定 `/prototype/candidate-profile`、公開LP `/prototype/lp-full`。

## 4. 実装順序（Phase）
- **Phase 1：画面整理とMVP化（プロトで完了）** — メニュー5整理・ダッシュボードをトップ・不要機能を後回し・改名。
- **Phase 2：政策をつくる** — 政策一覧/作成・AI政策秘書・**1枚政策カード**（タイトル/課題/解決策/効果/必要な支援/進捗/シェア短文）・政策12項目。
- **Phase 3：支援者CRM** — 一覧/登録・接触履歴・次回アクション・**今日連絡すべき支援者の抽出**・連絡文案。
- **Phase 4：政策応援ページ** — 作成・目標/支援額管理・成果報告作成。
- **Phase 5：公開ページ** — 候補者公開ページ・政策ページ・成果報告ページ。

## 5. Supabase テーブル（このSQLを実行・MVP優先）
```sql
create table candidates (
  id uuid primary key references auth.users on delete cascade,
  name text, party text, region text, title text, biography text, achievements text,
  sns_links jsonb, profile_image text, main_policies text[], support_needs text[],
  updated_at timestamptz default now()
);
create table policies (
  id bigint generated always as identity primary key,
  candidate_id uuid references candidates, title text, issue text, solution text,
  target_region text, current_data text, cause text, budget text, steps text,
  stakeholders text, expected_impact text, risks text, public_explanation text,
  status text, is_published boolean default false
);
create table supporters (
  id bigint generated always as identity primary key,
  candidate_id uuid references candidates, name text, organization text,
  email text, phone text, sns text, support_types text[], interested_policies text[],
  contact_history jsonb, next_action text, next_contact_date date, status text, memo text
);
create table policy_support_pages (
  id bigint generated always as identity primary key,
  candidate_id uuid references candidates, policy_id bigint references policies,
  title text, purpose text, target_amount int, current_amount int,
  supporter_count int, use_of_funds text, progress text, status text, is_published boolean default false
);
create table donations (
  id bigint generated always as identity primary key,
  page_id bigint references policy_support_pages, supporter_id bigint references supporters,
  amount int, date date, purpose text, name text, address text, occupation text,
  contact text, disclosure_status text, receipt_file text, memo text
);
create table tasks (
  id bigint generated always as identity primary key,
  candidate_id uuid references candidates, type text, title text, description text,
  due_date date, priority text, status text, related_supporter_id bigint, related_policy_id bigint
);
create table ai_generated_content (
  id bigint generated always as identity primary key,
  candidate_id uuid references candidates, type text, source_id bigint,
  prompt text, content text, status text, created_at timestamptz default now()
);
-- 各テーブルに RLS を有効化し「candidate_id = auth.uid() の行のみ読み書き可」
```
- Storage バケット: `avatars`（顔写真）、`receipts`（証憑）。
- ※ AI出力は `ai_generated_content` に「下書き(status=draft)」で保存し、公開・送信は status を変える運用に。

## 6. 保存処理の差し替え（中心作業）
- プロトの全画面は保存を **`app/prototype/_store.ts` の loadJSON / saveJSON** 経由で実施。
- このファイルの中身を Supabase の select/upsert に書き換えるだけで全画面が本番DBへ繋がる（呼び出し側は変更不要）。
- `_store.ts` 内に **保存キー→テーブル対応表** を記載。

## 7. ログイン・ファイル・デプロイ
- ログイン: 既存 `app/(auth)/login`・`sign-up` を Supabase Auth に接続。`auth.uid()` を `candidate_id` に。本番は `/prototype` の認証バイパス（`providers/auth-provider.tsx` の `AUTH_PATHS`）を戻す。
- ファイル: 顔写真→`avatars`、証憑→`receipts` にアップロードしURL保存。
- デプロイ: Vercel に接続→環境変数（`NEXT_PUBLIC_SUPABASE_URL` / `NEXT_PUBLIC_SUPABASE_ANON_KEY`）→push で自動公開。

## 8. コンプライアンス（必須表示）
- 全体: 「本サービスは政治活動・選挙運動・献金管理を支援するものです。実際の公開・送信・献金募集・収支報告にあたっては、政治資金規正法、公職選挙法、各自治体選挙管理委員会のルールを確認してください。」
- 投稿/配信作成時、献金管理時にも注意文を表示。
- **寄付・献金の実決済は今回入れない**（記録・表示のみ）。個人情報は RLS で本人のみアクセス。

### 補足：選挙候補者ロスターのアップロードは admin（運営）側
有権者が比較するための「選挙ごとの候補者一覧」は、**運営 admin でCSV/手動アップ→審査→承認→公開**する設計（政治家本人は自分の情報のみ充実）。本リポジトリ（政治家側）の対象外。
